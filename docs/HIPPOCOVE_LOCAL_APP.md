# Hippocove 本地打开说明

这份不是给已经熟悉仓的人看的。

它更像一张“把灯点起来”的小纸条，给第一次拿到仓的人，或者给会替人类指路的辅助 AI。

## 先说最省事的打开法

### macOS

直接在仓根目录双击：

- `00_双击启动_Hippocove.command`

它会做三件事：

1. 检查后端依赖是否已经装好
2. 自动在本机拉起 Hippocove 后端
3. 打开前台页面 `http://127.0.0.1:3460/`

### Windows

直接在仓根目录双击：

- `00_双击启动_Hippocove.cmd`

它会做同样三件事：

1. 检查后端依赖是否已经装好
2. 自动在本机拉起 Hippocove 后端
3. 打开前台页面 `http://127.0.0.1:3460/`

跑起来以后，旧实验台在：

- `http://127.0.0.1:3460/legacy/index.html`

## 如果想停掉本地后端

双击仓根目录里的停止脚本：

- macOS：`00_停止_Hippocove.command`
- Windows：`00_停止_Hippocove.cmd`

它会按仓里记录的 PID 把 Hippocove 本地后端停掉。

## 页面里的两个关键入口

### 1. 重置本地工作台

前台右上角和旧实验台右上角都带了：

- `重置本地工作台`

这不是只擦浏览器表面。

它会一起清掉：

- 本机 API 方案
- 提取缓存
- scope 运行记录
- Obsidian staging 导出草稿

如果你只是想从一张干净桌子重新开跑，就点它。

### 2. 隐私筛查

这一步还保留人工把关位。

也就是说：

- 记忆整编：已经适合继续让流程自动推进
- 隐私筛查：仍然建议人类最后看一眼、手动改一笔

## 它是不是只认 mac

不是。

真正跨平台的是这几层：

- Node.js 后端
- 浏览器前台
- 旧实验台
- `data/runtime_save` / `output/obsidian_staging` 这套本地目录结构

真正偏系统的，只是“仓里自带的双击钥匙”：

- `.command` 是给 macOS 的
- `.cmd` 是给 Windows 的

## 给辅助 AI 的一句人话口径

如果有人问“这个仓怎么用”，比较稳的说法是：

> 先双击仓根目录里对应系统的启动脚本，把工位点亮；前台跑主流程，旧实验台做整编和隐私筛查。如果要彻底清空本机缓存和 API 方案，就点页面右上角的“重置本地工作台”。

## 这东西现在还不是什么

它现在已经更像一套完整的本地工位了，但还不是那种封装好的 `.app` 图标应用。

所以更准确地说：

- 它已经有“仓内自带启动钥匙”
- 但还不是“完全不需要看仓目录”的成品 GUI 应用

再说死一点：

- 现在已经做到“下载后能找到清楚的双击入口”
- 但用户机器上仍然需要先装 `Node.js 20+`

如果要做到“什么都不会的人，机器上也什么都不用先装，双击就跑”，下一步就不是继续补脚本，而是打包成 Electron / Tauri 这类真正的桌面应用。

## 桌面应用这条线现在到哪

这仓已经补上了 Electron 桌面壳骨架：

- `main.mjs`
- 根目录 `package.json`

这层壳会：

- 打开桌面窗口
- 自动拉起 Hippocove 本地后端
- 把运行时数据写进应用用户目录，而不是源码目录

如果你是开发者，后续可以这样继续：

1. 在仓根目录跑 `npm install`
2. 本地调试：`npm run dev`
3. Windows x64 便携版 `exe`：`npm run pack:win`
4. Windows 安装器：`npm run pack:win:installer`
5. mac 安装包：`npm run pack:mac`

默认产物会落在仓根目录的 `dist/` 里。当前最适合直接发人的，是 `npm run pack:win` 打出来的 Windows x64 便携版 `exe`。

这代表它已经开始从“脚本启动的本地工位”往“可打包桌面应用”长过去了。
