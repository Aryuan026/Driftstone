# Hippocove 手测地图

这份不是技术设计稿。

它更像一张走位图，留给第一次接手这套仓的人：

- 先看哪里
- 先点哪里
- 怎么判断现在看到的是“正常实验态”，还是“真的有 bug”

## 1. 先确认你在看什么

Hippocove 现在是三层工位：

- `docs/index.html`
  GitHub Pages 首页。先看它讲得清不清楚。
- `index.html`
  正式前台。像入口台和等待台。
- `legacy/index.html`
  旧实验台。像后面的工坊，适合核对明文结果、调参、看批次。

所以手测时，不要只盯一张网页。
前台和旧实验台是两种位置。

## 2. 最小启动

```bash
cd server
npm install
npm run start
```

默认本地入口：

- 前台：`http://127.0.0.1:3460/`
- 旧实验台：`http://127.0.0.1:3460/legacy/index.html`
- 健康检查：`http://127.0.0.1:3460/api/health`

如果你只是想先判断“仓有没有断头”，先看：

- `/api/health`
- `/api/meta/routes`

这两只门能开，说明主骨架大概率还在。

## 3. 第一次手测建议顺序

### 第一步：先看首页是不是像入口，不像调试台

打开：

- `http://127.0.0.1:3460/`

先看三件事：

- API 设置是不是读得到方案
- 上传原始记录是不是说明清楚
- 底部星图是不是只是陪跑，不抢主流程

如果前台一上来就像在给开发者看日志，那不是你不会用，是这层还没收干净。

### 第二步：去旧实验台看人格工位

打开：

- `http://127.0.0.1:3460/legacy/index.html`

重点看：

- `人格工位`
- `API 测试`
- `MEMO 处理器`

这里要确认的是：

- 人格卡和语言指纹是不是明文可见、可改、可导出
- API 方案是不是能保存、切换、测试
- 旧工作台结果是不是能直接肉眼核对

如果这些地方还是黑箱，这套仓就还没到“适合别人调”的状态。

### 第三步：看主卡生长页

旧实验台里切到：

- `主卡生长`

重点不是“有没有漂亮动画”，而是：

- 当前 scope 是不是明确
- 原料包、最近动作、逐个递送是不是看得懂
- 已长出的卡是不是能看到
- 批量导出是不是通

如果这里能看见：

- 已入库主卡
- 当前草稿
- 清空当前草稿 / 清空已入库主卡
- 批量导出 Markdown

那就说明这层已经不是只给开发者看的了。

### 第四步：确认导出

导出不是最后才看的一步。

因为对这套仓来说，真正交出去的不是网页，而是：

- Obsidian Markdown
- Trace
- discard report
- staging 结构

所以至少要确认：

- 导出的 Markdown 路径是清楚的
- 目录结构跟约定一致
- 不是只在页面上“看起来成功”

## 4. 看到这些，不一定是 bug

### A. 前台默认是空桌

如果没有跑真实材料、没有同步人格桌面，首页看到的是：

- `Companion / You`
- 空的人格卡
- 空的语言指纹

这不是断头，是公开版默认状态。

### B. 旧实验台仍然比前台更重要

这是有意保留的。

如果你觉得“为什么很多真动作还在旧实验台里”，不是你理解错了，而是这版本来就把旧实验台保留成调参工坊。

### C. 味道不一样，不等于坏了

模型温度、人格口癖、记忆卡语气，这些已经进入个性化区间。

这份公开版交出去时，本来就不是要把“味道”锁死，而是把路和工位交出去。

## 5. 真正要警惕的 bug

这些才更像“断头”：

- `/api/health` 不通
- `/api/meta/routes` 缺主路由
- 页面能打开，但保存/导出完全不落盘
- 主卡生长看板永远空白，后台明明有卡
- 共享人格工位页面里有，后端读不到
- Trace / discard report 丢失

一句话说：

如果只是文风不对、UI 还丑、默认值还空，那叫实验态。
如果门能开但东西永远不写进去，那才叫 bug。
